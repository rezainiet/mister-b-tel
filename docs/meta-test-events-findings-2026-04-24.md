# Meta Test Events findings recorded on 2026-04-24

From Meta's Conversions API documentation, the platform states that a request sent with `test_event_code` should make the server event appear in the Test Events window once the request is sent. The same documentation also states that the Test Events tool generates a test ID and that the value must be sent as `test_event_code` in the payload.

From Meta's Business Help troubleshooting page, Meta says that if events do not appear within one minute, the recommended checks are browser-side troubleshooting steps such as confirming internet connectivity, disabling ad blockers, refreshing the site, and trying another supported browser. The troubleshooting note does not describe any special handling for server-side `Subscribe` events.

These findings suggest two things. First, `events_received: 1` remains strong evidence that Meta accepted the event payload. Second, Meta's public troubleshooting guidance does not document an event-type-specific exclusion for `Subscribe`, which keeps UI filtering, latency, or payload-shape differences as live possibilities to test further.

A second Meta reference confirms that `Subscribe` is the standard event for the start of a paid subscription, and Meta's example includes `value`, `currency`, and `predicted_ltv` in the event parameters. The same reference lists those fields directly in the Subscribe example, unlike simpler events such as PageView or ViewContent.

This does not by itself prove that those fields are strictly required for delivery, but it does establish a plausible payload-shape difference between the currently accepted server-side Subscribe test and Meta's own canonical Subscribe example. That makes an enriched isolated Subscribe test the next clean diagnostic step.

The user confirmed that the enriched isolated test event `isolated_server_subscribe_enriched_1777052853` appeared in Meta Test Events as a Subscribe event. This sharply narrows the practical cause: Meta accepted the earlier minimal Subscribe payload, but the Subscribe event became visible in the Test Events UI once the payload was shaped like Meta's own canonical Subscribe example, with `value`, `currency`, and `predicted_ltv` present in `custom_data`.

Operationally, the minimal production fix is to add those canonical subscription fields to the server-side Subscribe payload generated by `fireSubscribeEvent()`.

Desktop CTA clickability diagnosis on the published landing showed that the visible `GROUPE TELEGRAM` anchor is present, has `pointer-events: auto`, has no disabled state, and occupies a normal clickable rectangle. Its current `href` is a direct custom protocol deep link of the form `tg://resolve?domain=Misternb_bot&start=...`, with `target="_self"`.

This strongly suggests that the laptop problem is not an overlay blocking clicks. Instead, the visible CTA is clickable in the DOM but points to a `tg://` custom scheme that is fragile on desktop browsers and laptops, especially when no Telegram protocol handler is available or the browser declines to open the custom scheme. The correct fix is likely a desktop-safe fallback path rather than a z-index or pointer-events change.

Post-fix validation on the dev preview confirmed that the visible `GROUPE TELEGRAM` CTA now resolves to `https://t.me/Misternb_bot` with `_self` on desktop/laptop rendering. This replaces the earlier desktop-fragile `tg://resolve?...` target and confirms the laptop-facing clickability fix is active in the updated preview build.
A final controlled comparison on 2026-04-24 closed the remaining uncertainty. While the user watched the Test Events panel live, three fresh server-side tests were sent: one isolated `PageView` using a realistic browser fingerprint, one isolated `PageView` using the earlier synthetic `MetaTest/1.0` fingerprint, and one batched `PageView` plus `Subscribe` request matching the earlier disputed manual format. The user confirmed visibility for all three cases, including the batched request.

This rules out the main alternate explanations that were still open. The accepted-but-not-seen manual send was not caused by a wrong pixel view, not caused by the synthetic fingerprint alone, and not caused by batching `PageView` plus `Subscribe` into one request. The practical cause is that Meta Test Events behaves like a short-lived live stream: an accepted event can still be missed from the visible panel if it is not checked immediately after sending.

The operational fix for the published Mister B landing is therefore not to restore browser-side Meta delivery. Instead, the minimal reliable fix is to restore a server-only `meta_test_code` passthrough so that explicit test visits can be routed into Meta Test Events on demand. The landing now captures an optional `meta_test_code` query parameter, forwards it to the server-side `PageView` call, embeds it into the Telegram start payload, and reuses it when the Telegram `/start` webhook fires the server-side `Subscribe` event. In production, ordinary visits still remain server-side only and do not send test-event codes unless the testing query parameter is intentionally present.
